## Why this matters

`.hugo_build.lock` is generated by Hugo and already listed in `.gitignore`, but it remains in Git's index from an earlier commit. This creates a misleading tracked artifact and allows local server activity to appear as repository state. Removing the index entry aligns Git behavior with the documented ignore policy.

## Current state

- `.gitignore:4` contains `.hugo_build.lock`.
- `git ls-files .hugo_build.lock` prints `.hugo_build.lock`, proving it is still tracked.
- The file is generated/empty state, not application source.
